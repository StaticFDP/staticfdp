StaticFDP Ecosystem

FAIR Data Points
without a server

Three open-source reference implementations — a FAIR Data Point, an FDP Index, and a Virtual Platform — all deployable on static hosting with no dedicated server required.

The ecosystem

The StaticFDP ecosystem provides static-hosting reference implementations for the three layers of the FAIR Data Point architecture. Each component is a standalone repository that can be deployed independently on GitHub Pages, Codeberg Pages, or any static host — and every component lives on both GitHub and Codeberg for sovereignty and redundancy.

staticfdp

Static FAIR Data Point

A template for publishing a DCAT-conformant FDP using only static hosting. Contributors submit data via web forms; a CI/CD pipeline converts every submission to RDF Turtle and updates the FDP automatically.

  • Form → GitHub / Forgejo Issues → RDF
  • ORCID authentication via Cloudflare Worker or Deno
  • GitHub Actions and Woodpecker CI pipelines included
  • Configurable: GitHub, Codeberg, or both
GitHub → Codeberg →
staticfdp-index

Static FDP Index

A static FDP Index that lets FAIR Data Points register themselves and be discoverable. Registration is an Issue or a YAML pull request; a CI pipeline periodically harvests all registered FDPs and rebuilds the index catalog as RDF + HTML.

  • FDPs register via Issues or YAML PR
  • Harvest pipeline fetches each FDP's catalog.ttl
  • Serves DCAT index + HTML discovery page
  • SPARQL-over-static demo included
GitHub → Codeberg →
staticfdp-vp

Static Virtual Platform

A static Virtual Platform that aggregates multiple FDP Indexes into a single federated discovery hub. Data stays at its source; the VP only stores pointers. A CI pipeline rebuilds the aggregated catalog on a schedule.

  • Aggregates N FDP Indexes into one catalog
  • HTML search + browse interface — no server
  • DCAT-AP federation graph served as Turtle + JSON-LD
  • Scheduled rebuild via GitHub Actions / Woodpecker CI

How the layers connect

Each layer is independently deployable. Together they form a fully FAIR-compliant federation — from individual dataset submissions up to cross-institutional discovery.

GitHub · Codeberg · or both

Every component in the ecosystem is designed to run on either platform — or both simultaneously for data sovereignty and redundancy.

GitHub (US)

GitHub Pages for static hosting. GitHub Actions for CI/CD. GitHub Issues as the data store for submissions.

Codeberg (EU)

Codeberg Pages on Hetzner Frankfurt. Woodpecker CI for pipelines. Forgejo Issues as the data store. GDPR-friendly jurisdiction.

Both (dual-write)

Form submissions go to both platforms simultaneously. Either platform can serve data independently. Run scripts/setup.sh to configure.

Quick start

  1. Fork the component you need (staticfdp, staticfdp-index, or staticfdp-vp)
  2. Run bash scripts/setup.sh — choose GitHub, Codeberg, or both
  3. Set the required secrets (GITHUB_TOKEN, ORCID_CLIENT_ID, etc.)
  4. Enable GitHub Pages (branch: main, path: /docs)
git clone https://github.com/StaticFDP/staticfdp
cd staticfdp
bash scripts/setup.sh
